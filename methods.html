Methods · under the hood

How the pipeline works

One consolidated, leakage-safe, thermal-safe pipeline takes pre-extracted UNI2 patch embeddings through aggregation, classical classification, and per-cancer survival analysis. The essentials are below; deeper rationale lives in the expandable panels.

Data

Two embedding cohorts are supported side-by-side. The original CHOL cohort: 59 WSIs (39 tumor, 20 normal) across 44 participants, with 15 patients contributing matched tumor + normal pairs. The expanded UNI2 cohort: 2 075 WSIs across 1 850 participants spanning 8 TCGA cancer projects (ACC, CESC, CHOL, COAD, ESCA, LIHC, READ, STAD), of which 1 608 have valid survival data (557 events). Patch-level embeddings are 1 536-dim per patch with variable patches per slide, pre-extracted with the UNI2 foundation model.

Slide-level aggregation

Each variable-length patch matrix (N × 1536) is aggregated into a fixed slide vector. The default "rich" aggregation concatenates six summary statistics — mean, std, P10, P25, P75, P90 — yielding a 9,216-dim representation. The pipeline also supports plain mean aggregation (1,536-dim).

Models

Classification models

10 laptop-friendly classifiers, single-threaded by default:

  • Linear: LogisticRegression (L1, L2), RidgeClassifier, LinearSVC
  • Ensembles: RandomForest, ExtraTrees, GradientBoosting
  • Other: KNN-5, LinearDiscriminantAnalysis, GaussianNB

Preprocessing pipeline: StandardScaler → PCA (95 % variance retained) → classifier. Preprocessing is refit inside every CV fold to prevent data leakage.

Survival models

Six survival estimators evaluated side-by-side per cancer type:

  • Cox_AgeSex — Cox baseline on age + gender (clinical-only floor)
  • Cox_WSI_plus_Clin — Coxnet on WSI PCs ⊕ clinical covariates (combined model)
  • CoxnetLasso / CoxnetElasticNet — penalized Cox on WSI PCs
  • RSF — Random Survival Forest (n_estimators = 100, max_depth = 5)
  • GradientBoostSurv — gradient-boosted survival

All survival models are fit on PCA-reduced features (95 % variance retained, refit on each cancer subset). Clinical covariates are standardized separately so they sit on the same scale as the PCs in the combined model.

Cross-validation

Classification auto-selects a leakage-safe split: when any participant has multiple slides (e.g. matched tumor + normal), GroupKFold or LeaveOneGroupOut is used with participant ID as the group. No participant ever appears in both train and test.

Survival uses pooled leave-one-participant-out. For each participant, the model is fit on the other N − 1 and used to predict that patient's risk. All N held-out risks are pooled into a single vector and one concordance_index_censored is computed. To get a 95 % CI on that single number, 1 000 bootstrap resamples of the (event, time, risk) triples are taken; the 2.5 / 97.5 percentiles are reported as ci_low / ci_high.

Why pooled LOPO instead of 5-fold for survival

Per-fold C-index requires comparable event pairs in the test set. With 55 patients and 27 events split 5 ways, some folds had ≈ 5 events — too few to estimate C-index stably. The previous RSF = 0.71 ± 0.22 on CHOL reflected fold-split noise, not predictive power. Pooled LOPO uses every event exactly once across the test set and yields a single stable C-index whose uncertainty is honestly captured by the bootstrap CI.

What each survival output row also records
  • a per-SD hazard ratio with 95 % CI and p-value (univariate lifelines Cox on the standardized risk score) — quantifies the strength of the continuous association;
  • a log-rank p-value from a median split of the predicted risk — the standard "does this model stratify into clinically distinct groups?" test;
  • feature_mode and clinical_covariates_used columns so the WSI-only, clinical-only, and combined rows are directly comparable.

Permutation test (classification)

To check whether AUC = 0.80 was a small-N fluke, the grade labels were randomly shuffled 200 times. For each shuffle, the same LOO-cross-validated KNN-5 pipeline was run end-to-end and the resulting AUC recorded. The empirical p-value is the fraction of null AUCs ≥ the baseline, with a +1 / N+1 correction (Phipson & Smyth 2010).

Thermal-safe design

The pipeline was built around a laptop that overheats under sustained load. H5 files are loaded one at a time with explicit cooldown pauses; models default to n_jobs = 1; progress is checkpointed every 5 files so a crash never costs more than ~30 seconds of work.

  • H5 files loaded one at a time with explicit cooldown pauses
  • n_jobs = 1 by default for all parallelisable models
  • Checkpointing every 5 files (recover with --resume)
  • --cool mode for extended pauses between files

Reproducibility

The site is the readable front-end; everything below regenerates the underlying results. All commands are in wsi_survival_pipeline.py.

Setup 
pip install numpy pandas scipy scikit-learn h5py matplotlib seaborn
pip install scikit-survival lifelines
Reproduce the validated grade finding (CHOL) 
python wsi_survival_pipeline.py \
    --aggregated-csv results_lightweight/aggregated_embeddings.csv \
    --output-dir results_v2 \
    --target grade --no-survival
Pan-cancer survival run (UNI2 cohort) 
# Full pipeline: aggregation + classification + per-cancer survival LOPO.
# Auto-discovers both embedding directories.
python wsi_survival_pipeline.py \
    --clinical-data ./CLINICAL_FULL.parquet \
    --output-dir ./results_v2 --cool

# Resume after a crash (uses the on-disk checkpoint)
python wsi_survival_pipeline.py --resume --output-dir ./results_v2

# Survival-only re-run from a cached aggregated CSV
python wsi_survival_pipeline.py \
    --aggregated-csv ./results_v2/aggregated_embeddings.csv \
    --output-dir ./results_v2 --no-classification
Other classification targets 
# Tumor vs normal (saturates at AUC = 1.0 on UNI2 — useful only as a sanity check)
python wsi_survival_pipeline.py --target tumor_normal

# Mutation status (binary mut vs wt — needs ./TCGA_data/MUTATIONS_FULL.parquet)
python wsi_survival_pipeline.py --target mutation:BAP1

# Survival-quartile classification (short Q1 vs long Q4 — classification, not survival CV)
python wsi_survival_pipeline.py --target survival_quartile
Thermal & runtime controls 
--cool                      # aggressive cooldowns (2 s between H5 files)
--n-jobs 1                  # single-threaded (default; raise for hotter hardware)
--bootstrap-iters 1000      # iterations for survival CI (lower = faster, wider CI)
--min-survival-patients 30  # drop cancers smaller than this from survival
--resume                    # pick up aggregation from .aggregation_checkpoint.json
